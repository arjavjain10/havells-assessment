# Havells Customer Voice Intelligence Agent

Turns raw, messy product reviews into grounded answers a PM can act on:
what people are unhappy about, on which product, and whether it's getting
better or worse over time — with every claim traceable back to a specific
review.

## Architecture

Three agents, deliberately kept separate rather than one monolithic prompt,
so each piece is independently testable and swappable:

```
reviews.csv
     │
     ▼
┌─────────────────────┐   per review → structured JSON
│  Extraction Agent    │   {aspect, sentiment, evidence}
│  (LLM or rule-based) │   evidence = exact substring from the review
└─────────┬────────────┘
          ▼
┌─────────────────────┐   deterministic, NO LLM
│  Aggregation Agent   │   groups by product × aspect × month
│  (pure code)         │   computes mention-rate trend (rising/falling/stable)
└─────────┬────────────┘
          ▼
┌─────────────────────┐   question → retrieve matching (product, aspect)
│  Grounded QA Agent   │   slice + raw evidence → LLM answers ONLY from
│  (retrieval + LLM)   │   that context → post-hoc check that every number
└─────────────────────┘   in the answer matches the aggregation table
```

**Why this split:**
- The **Extraction Agent** is the only place unstructured text gets touched.
  Every aspect it emits carries an `evidence` string that must be an exact
  substring of the source review — anything the LLM can't point to in the
  text is dropped before it ever reaches downstream stages.
- The **Aggregation Agent** is plain code, not an LLM call. Numbers and
  trend directions need to be reproducible and auditable, not something
  that can vary between two LLM calls on the same data.
- The **QA Agent** never lets the model answer from its own knowledge. It
  retrieves the relevant slice of the aggregation table + a few evidence
  snippets, hands only that to the LLM, and then re-checks the LLM's
  answer: every percentage/number quoted must match a number that's
  actually in the retrieved data, or the answer gets flagged. If nothing
  relevant is retrieved at all (e.g. a question about a product/issue not
  in the data), the agent says so instead of guessing.

## Dataset

Synthetic, generated by `data/generate_data.py` — 240 reviews across three
Havells product lines (fans, water heaters, mixer grinders), Jan–Jun 2026.
Three trends are seeded on purpose so the pipeline's output can be checked
against a known answer:

| Product | Aspect | Seeded pattern |
|---|---|---|
| Havells Fan X1 | motor noise | rises steadily month over month |
| Havells Instanio Water Heater | thermostat | high, then drops sharply after March (simulated fix) |
| Havells Mixer Grinder MG100 | blade durability | stays flat (control, no real trend) |

This isn't cheating — it's what makes the evaluation section meaningful in
an hour: instead of eyeballing plausibility, I can check the pipeline
actually recovers patterns I know are there. Swap in a real dataset (e.g.
Amazon/Flipkart appliance reviews, or any Kaggle product-review set) by
replacing `data/reviews.csv` with the same columns: `review_id, product,
date, month, rating, review_text`.

## How to run

```bash
cd havells_review_agent
pip install -r requirements.txt   # google-genai, only needed for real LLM calls

# Optional: use real Gemini calls for extraction + QA
export GEMINI_API_KEY=AIza...
# (GOOGLE_API_KEY also works, and GEMINI_MODEL defaults to gemini-2.5-flash —
#  override it, e.g. export GEMINI_MODEL=gemini-2.5-pro, if you want)

# Generate data (auto-runs on first use anyway) + run pipeline
python main.py --build

# Ask a single question
python main.py --ask "Is the fan getting noisier over time?"

# Or interactive mode
python main.py
```

Without `GEMINI_API_KEY`/`GOOGLE_API_KEY` set, everything still runs end to
end using rule-based fallbacks (keyword aspect extraction, templated but
still fully grounded answers) — useful for a quick demo with no API access.
With the key set, extraction and answering go through Gemini
(`gemini-2.5-flash` by default) with the same grounding checks enforced on
top: extracted "evidence" must be an exact substring of the review, and
every number in a QA answer must match the aggregation table or it gets
flagged.

Run the evaluation suite after building:
```bash
python eval.py
```

## Evaluation

`eval.py` checks three things, matching what actually matters for this
system:

1. **Extraction accuracy** — precision/recall of extracted (aspect,
   sentiment) pairs against the seeded ground truth (`data/ground_truth.json`).
2. **Trend detection accuracy** — does the aggregation agent recover all
   three seeded patterns correctly (rising / falling / stable)?
3. **QA groundedness** — for a fixed set of test questions, is every
   number in the answer traceable to the aggregation table, and is an
   out-of-scope question (e.g. about a product/issue not in the data)
   correctly answered with "no supporting data" rather than a guess?

In production this would extend to: a held-out human-labeled sample of
reviews for extraction QA on a rolling basis, and a "trap" question set
that's regenerated periodically to catch grounding regressions before they
reach PMs.

## Scaling to the full catalogue

- **Extraction** is embarrassingly parallel — one review at a time, no
  shared state. Move it behind a queue (SQS/Kafka) with workers scaling
  horizontally; this is the dominant cost driver so it's worth batching
  LLM calls where the provider supports it.
- **Aggregation** becomes a scheduled batch/streaming job (e.g. daily) per
  product line rather than an in-memory recompute; store results in a
  proper time-series/OLAP store (e.g. a Postgres table keyed by product ×
  aspect × month, or a column store) instead of a JSON blob.
- **QA retrieval** currently does keyword matching over a small in-memory
  table — fine for one product, not for a full catalogue. At scale, swap
  in a vector index (aspect descriptions + evidence snippets embedded) so
  retrieval generalizes beyond exact keyword overlap, while keeping the
  same "only answer from retrieved context + verify numbers" contract.
- **Aspect taxonomy drift**: with one product, a fixed aspect list is fine.
  Across a catalogue, aspects need to be clustered/normalized (e.g.
  "motor noise" vs "loud motor" vs "buzzing sound" should collapse to one
  aspect) — this becomes its own periodic clustering step between
  extraction and aggregation.

## File structure

```
havells_review_agent/
├── README.md
├── main.py                      # orchestrator + CLI
├── eval.py                      # evaluation harness
├── data/
│   ├── generate_data.py         # synthetic dataset generator
│   ├── reviews.csv              # generated dataset (input)
│   └── ground_truth.json        # seeded labels, for eval only
├── agents/
│   ├── extraction_agent.py      # review text -> structured aspects
│   ├── aggregation_agent.py     # structured facts -> trends (pure code)
│   └── qa_agent.py              # grounded retrieval + Q&A
└── outputs/
    ├── extractions.json         # cached extraction output
    └── aggregation.json         # cached aggregation table
```
